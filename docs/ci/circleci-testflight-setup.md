# CircleCI → TestFlight setup (non-interactive EAS Build + Submit)

This repo ships a CircleCI pipeline (`.circleci/config.yml`) that, on every `v*`
git tag, builds the iOS app on **Expo EAS** and submits it to **TestFlight** —
**fully non-interactively**. You never run `eas` locally, never answer an Apple
login prompt, and no secret is committed. CircleCI only drives `eas-cli` on a
cheap Linux box; the macOS build and code signing happen on EAS's cloud, and iOS
signing credentials are generated by EAS from an **App Store Connect API key**.

All one-time work below is web-portal clicks (Expo, Apple) plus pasting values
into CircleCI env vars. Do it in order.

## 1. Expo account + access token

1. Create a free account at https://expo.dev. Your username is the `EXPO_OWNER`
   value.
2. Create an access token: expo.dev → **Account → Settings → Access Tokens** →
   create. Save it — it becomes the CircleCI `EXPO_TOKEN`.

You do **not** run `eas login` or `eas init` locally — the bootstrap CI job
(step 4) creates the EAS project for you.

---

## 2. Apple: bundle id, app record, API key, team id

You need a paid **Apple Developer Program** membership.

1. **Bundle id.** Choose a unique iOS bundle identifier (e.g.
   `com.yourname.hydra`). This is `IOS_BUNDLE_ID`. It is permanent once the app
   record exists.
2. **App record.** App Store Connect → **My Apps → + → New App**, select/enter
   your bundle id, name it. On the app's **App Information** page note the
   numeric **Apple ID** — that's `ASC_APP_ID`.
3. **App Store Connect API key.** App Store Connect → **Users and Access →
   Integrations → App Store Connect API** → generate a key with the **App
   Manager** role. Download the `.p8` (one-time download), and note the **Key
   ID** (`EXPO_ASC_KEY_ID`) and **Issuer ID** (`EXPO_ASC_ISSUER_ID`). This single
   key lets EAS both create signing credentials and upload to TestFlight.
4. **Team id + type.** developer.apple.com → **Membership** → note the **Team
   ID** (`EXPO_APPLE_TEAM_ID`). For a personal account `EXPO_APPLE_TEAM_TYPE` is
   `INDIVIDUAL` (otherwise `COMPANY_OR_ORGANIZATION`, or `IN_HOUSE` for
   enterprise).

Base64-encode the `.p8` for CircleCI (it stores single-line values):

```bash
base64 -i AuthKey_XXXXXXXXXX.p8 | pbcopy   # now in your clipboard → ASC_API_KEY_BASE64
```

## 3. CircleCI: set up project + environment variables

1. In CircleCI, **Set Up Project** for this repo (use the existing
   `.circleci/config.yml`).
2. Create a **context** named `hydra-cci` (Organization Settings → **Contexts**
   → Create Context) and add the variables below to it. The workflows reference
   this context, so the vars are shared across jobs rather than set per-project:

   | Variable | Value |
   |---|---|
   | `EXPO_TOKEN` | Expo access token (step 1.2) |
   | `EXPO_OWNER` | your Expo username (step 1.1) |
   | `EAS_PROJECT_ID` | filled after step 4 (leave unset for now) |
   | `IOS_BUNDLE_ID` | your iOS bundle id (step 2.1) |
   | `ASC_API_KEY_BASE64` | base64 of the `.p8` (step 2) |
   | `EXPO_ASC_KEY_ID` | API Key ID (step 2.3) |
   | `EXPO_ASC_ISSUER_ID` | Issuer ID (step 2.3) |
   | `EXPO_APPLE_TEAM_ID` | Apple Team ID (step 2.4) |
   | `EXPO_APPLE_TEAM_TYPE` | `INDIVIDUAL` (or your team type) |
   | `ASC_APP_ID` | numeric App Store Connect app id (step 2.2) |
   | `ANDROID_PACKAGE` | optional, only if you build Android |
   | `SENTRY_AUTH_TOKEN` / `SENTRY_ORG` / `SENTRY_PROJECT` | optional, see notes |

> **Bundle id on the EAS build server.** `app.config.ts` reads `IOS_BUNDLE_ID`
> when EAS runs `prebuild` on its build server. EAS does **not** see CircleCI's
> env vars there, so also add `IOS_BUNDLE_ID` (and `ANDROID_PACKAGE` if used) as
> an **EAS environment variable**: expo.dev → your project → **Environment
> Variables** → environment **production**, plaintext. Otherwise prebuild falls
> back to the upstream `com.dmilin.hydra` default. (`EXPO_OWNER` /
> `EAS_PROJECT_ID` are only needed by eas-cli in CircleCI, not on the build
> server.)

## 4. Bootstrap your EAS project (one-time CI job)

This creates the EAS project under your account without any local tooling.

1. Push a branch named exactly `eas-bootstrap`:

   ```bash
   git switch -c eas-bootstrap && git push -u origin eas-bootstrap
   ```

2. CircleCI runs the `bootstrap` workflow → `eas init` creates the project and
   the job log prints the **projectId** (`eas project:info`).
3. Copy that id into the `EAS_PROJECT_ID` CircleCI env var (step 3). Delete the
   `eas-bootstrap` branch when done.

(If you ever already have a projectId, skip this and just set `EAS_PROJECT_ID`.)

---

## 5. Cut a release

1. Bump the app version in `package.json` (`version` drives the iOS marketing
   version; EAS auto-increments the build number — `eas.json` sets
   `appVersionSource: remote` + `production.autoIncrement`).
2. Tag and push:

   ```bash
   git tag v4.0.2
   git push origin v4.0.2
   ```

3. CircleCI runs the `release` workflow → EAS builds iOS, generates signing
   credentials from your ASC API key, and `--auto-submit` uploads to App Store
   Connect. After Apple finishes processing it appears in **TestFlight**; add it
   to a TestFlight group to reach testers.

## Notes & gotchas

- **No secrets in git.** The `.p8` is passed as `ASC_API_KEY_BASE64` and written
  to disk only during the build; the `submit.production.ios` block is injected
  into `eas.json` at runtime from env vars. The committed `eas.json` stays clean.
- **First release establishes signing.** The first `release` build is what makes
  EAS generate your distribution cert + provisioning profile (from the ASC API
  key). Subsequent builds reuse them.
- **Sentry plugin.** Without `SENTRY_AUTH_TOKEN` (+ `SENTRY_ORG` /
  `SENTRY_PROJECT`) the Sentry plugin **warns and skips** — it does not fail the
  build. Set them for your own crash reporting, or remove the plugin.
- **EAS Build cost.** The macOS build runs on EAS. The free tier covers 15 iOS
  builds/month on a low-priority queue; a paid Expo plan buys priority/volume.
- **Tags only.** The `release` workflow ignores all branches and runs only on
  `v*` tags. Change the `filters` in `.circleci/config.yml` to also build on
  `master` if you want a faster cadence.
